===================================================================
Adi — Fly secrets checklist
===================================================================

For each of Shane and Meg, run the `fly secrets set` commands below
against that user's app. Replace the placeholder values with real
tokens. Run these ONCE before first deploy; `fly secrets set` persists
them on Fly's side and they survive redeploys.

NEVER paste these commands with real values into git, chat, or any
persistent record. Run them in a terminal and clear the history.

-------------------------------------------------------------------
                        SHANE (adi-shane)
-------------------------------------------------------------------

# --- Required: gateway auth ---
# Generates a random 32-byte token; copy-paste the output back into
# a secrets set command, OR let openssl generate-and-set in one line:
fly secrets set OPENCLAW_GATEWAY_TOKEN=$(openssl rand -hex 32) -a adi-shane

# --- Required: LLM provider(s) ---
fly secrets set ANTHROPIC_API_KEY=sk-ant-REPLACE_ME -a adi-shane

fly secrets set SLACK_APP_TOKEN=xapp-REPLACE_ME -a adi-shane

# --- Telegram (Shane's BotFather bot) ---
fly secrets set TELEGRAM_BOT_TOKEN=REPLACE_ME -a adi-shane

# --- Todoist (Shane's API token) ---
# Todoist → Settings → Integrations → Developer → API token
fly secrets set TODOIST_API_TOKEN=REPLACE_ME -a adi-shane

# --- CLIProxyAPI (sidecar model gateway) ---
# These are OPTIONAL. If you omit them, the entrypoint generates random
# values on first boot and writes them into the volume's config.yaml,
# which is fine for single-machine use. Set them explicitly only if you
# want to control the values yourself (e.g. for rotation, or to share the
# management key with a local control-panel instance).
#
# fly secrets set CLIPROXY_API_KEY=$(openssl rand -hex 32) -a adi-shane
# fly secrets set CLIPROXY_MGMT_KEY=$(openssl rand -hex 32) -a adi-shane
#
# NOTE: OAuth logins (Claude.ai, ChatGPT, Gemini) are NOT set via Fly
# secrets — they're interactive flows performed ONCE via `fly ssh console`
# after first deploy. See DEPLOY.md §7 for the CLIProxyAPI setup flow.

# --- Supabase (Phase 1b; set now if the project is already provisioned) ---
# Use the "service_role" key for write access. Shane's Adi uses this
# key and relies on RLS to restrict to Shane's rows.
fly secrets set SUPABASE_URL=https://REPLACE_ME.supabase.co -a adi-shane
fly secrets set SUPABASE_SERVICE_KEY=eyJ...REPLACE_ME -a adi-shane

# --- Tailscale (family relay mesh: adi-meg → adi-shane CLIProxyAPI) ---
# Mint at: Tailscale admin → Settings → Keys → Generate auth key.
# REQUIRED settings for this key:
#   - Reusable: YES   (gateway restarts reuse the same node identity)
#   - Ephemeral: NO   (node survives across Fly machine restarts)
#   - Pre-approved: YES (skips admin approval on first join)
#   - Tags: tag:fly-gw
# Key looks like: tskey-auth-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
fly secrets set TAILSCALE_AUTHKEY=tskey-auth-REPLACE_ME -a adi-shane

# --- Cross-gateway operator token (Adi-on-Shane queries Meg's gw) ---
# Adi running on adi-shane holds Meg's OPENCLAW_GATEWAY_TOKEN so she can
# call Meg's gateway (http://adi-meg.<tailnet>.ts.net:3000) for status,
# session lookups, channel health, etc. Value MUST equal the
# OPENCLAW_GATEWAY_TOKEN secret set on adi-meg (see below).
# Set this AFTER you've set Meg's OPENCLAW_GATEWAY_TOKEN and have that
# value on hand.
fly secrets set ADI_MEG_GATEWAY_TOKEN=REPLACE_WITH_MEGS_OPENCLAW_GATEWAY_TOKEN -a adi-shane

# --- Supabase (same project; different RLS-scoped key recommended) ---
# You can use the same service_role key for both, but we recommend
# creating a second API key scoped to Meg via Supabase's new "Custom
# Claims" or an RLS-verifying JWT. For initial deploy, the same key
# is acceptable — RLS policies still enforce isolation by owner_id.
fly secrets set SUPABASE_URL=https://REPLACE_ME.supabase.co -a adi-meg
fly secrets set SUPABASE_SERVICE_KEY=eyJ...REPLACE_ME -a adi-meg

# --- Tailscale (family relay mesh) ---
# Mint a SECOND auth key for Meg's box — do NOT reuse Shane's key. Same
# settings as Shane's: reusable, non-ephemeral, pre-approved, tag:fly-gw.
# Separate keys give you independent revocation: if Meg's Fly machine is
# ever compromised, you can revoke JUST her tailnet membership without
# kicking Shane off.
fly secrets set TAILSCALE_AUTHKEY=tskey-auth-REPLACE_ME -a adi-meg

# --- CLIProxyAPI shared API key (must match Shane's box) ---
# Meg's openclaw routes model calls to Shane's CLIProxyAPI over the
# tailnet. Both sides must present the SAME api-key to authenticate.
# Required steps:
#   1. On Shane's box, grab the current api-key:
#        fly ssh console -a adi-shane
#        cat /data/cliproxy/current-api-key
#   2. Paste that value here as CLIPROXY_API_KEY.
# If you haven't deployed Shane yet, generate the key explicitly BEFORE
# Shane's first deploy:
#   fly secrets set CLIPROXY_API_KEY=$(openssl rand -hex 32) -a adi-shane
# and use the same value here.
fly secrets set CLIPROXY_API_KEY=REPLACE_WITH_SHANES_CLIPROXY_API_KEY -a adi-meg

# --- Shane's Tailnet IP (for cliproxy relay target) ---
# Meg's openclaw.meg.json has cliproxy.baseUrl = http://${SHANE_TAILNET_IP}:8317/v1
# to relay model calls to Shane's CLIProxyAPI over Tailscale. We use a raw
# tailnet IP (not MagicDNS) because Fly containers don't support Tailscale's
# DNS takeover ("getting OS base config is not supported"), and accept-dns=true
# partially breaks openclaw's undici dispatcher for non-tailnet traffic.
# Find Shane's IP: `fly ssh console -a adi-shane -C "tailscale --socket=/data/tailscale/tailscaled.sock ip -4"`.
# With non-ephemeral Tailscale keys + persistent /data/tailscale state, the IP
# is stable across redeploys. If Shane ever reinstalls cleanly or changes
# tailnets, update this secret.
fly secrets set SHANE_TAILNET_IP=100.111.92.71 -a adi-meg
